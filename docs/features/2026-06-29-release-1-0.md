# Release 1.0

> Historical delivery record. It may not describe the current system.

## Goal

Ship the coordinated 1.0 package line with deliberate publishing, structural packing, public API
normalization, Story-to-Tale naming, and retirement of obsolete packages.

## Context

The pre-1.0 surface contained inconsistent registration names, obsolete package identities, and a
hand-maintained release process vulnerable to omissions. Supported packages had independently
maintained versions, while Logging had already reached `1.1.1`. The workflow packed only a subset
of packable projects, omitted Hangfire and all testing companions, still included deprecated
projects, and published from every push to `master`.

The repository also carried a source-less but downloaded `ApiClient` package identity, incomplete
NuGet metadata, stale solution documentation entries, and two names for one orchestration framework:
Story infrastructure around a Tale authoring surface.

Release 1.0 was selected as the one honest point to make a coordinated compatibility break rather
than freezing those inconsistencies into a stable contract.

## Original decision

### Coordinated versions

Set one shared `1.0.0` default for supported packages. Logging used `1.2.0` to avoid a package
downgrade. Deprecated Scheduler and Guards projects stayed outside the shared supported line.

### Public registration naming

Rename public DI, middleware, and endpoint extensions to branded `AddSol*`, `UseSol*`, and
`MapSol*` forms without obsolete forwarders. Update core tests and DreamTravel in the same build-green
waves.

### Package retirement

Make `SolTechnology.Core.HTTP` the documented successor to the source-less ApiClient package. Stop
packing deprecated source projects and provide a manual, confirmation-gated workflow that enumerates
and unlists every published version of ghost package IDs.

NuGet server-side deprecation was initially planned but amended after confirming that no public CLI
command existed. Automated unlisting became the repository-owned mechanism; web-UI deprecation was
left as an optional manual follow-up.

### Publish hardening

Gate publishing behind a deliberate release trigger and discover every packable solution project
structurally instead of maintaining a list of pack commands. Centralize package metadata and SDK
SourceLink settings in `src/Directory.Build.props`.

### Story-to-Tale rebrand

Create `SolTechnology.Core.Tale` as a new package identity and apply the Tale noun to the package,
namespace, handler, manager, controller, repository, options, errors, builder, registry, instance,
and status types. Collapse the redundant nested Tale namespace. Retire the old Story package and
document its successor.

The complete rebrand was chosen because renaming only the package while retaining Story-prefixed
infrastructure would preserve the same two-noun confusion.

## Alternatives considered

### Keep obsolete forwarding methods

Rejected because old and new names would coexist permanently in IntelliSense and double the public
surface that 1.0 was intended to stabilize.

### Release 1.0 without renaming

Rejected because collision-prone names would become stable and require a future major release to
fix.

### Keep per-project versions

Rejected because it had already produced drift and an accidental Logging version ahead of the rest
of the package line.

### Hand-list package commands

Rejected because the existing list had silently omitted an entire production module and all testing
companions.

### Keep Story or perform a partial Tale rename

Rejected because the package, base classes, examples, and authoring DSL would continue to use two
nouns. A full Tale rebrand was selected despite being the widest migration.

## Scope

- Set supported packages to `1.0.0`, retaining Logging `1.2.0`.
- Rename registration and middleware APIs to `AddSol*`, `UseSol*`, and `MapSol*`.
- Rebrand Story to Tale.
- Pack all eligible solution projects and gate NuGet publishing.
- Add a manual package-unlisting workflow.

## Implementation plan

Deliver API renames and package moves, update samples and tests, centralize metadata, harden
publishing, and sweep documentation. The plan used a premortem verdict of Go with mitigations and
split the Tale rebrand into its own build-green wave.

## Acceptance criteria

- Supported packages build, test, and pack from the solution.
- NuGet publishing occurs only on tags or manual dispatch.
- Obsolete packages can be explicitly unlisted.
- Current documentation uses supported public APIs.
- Supported package versions come from one central default with explicit exceptions.
- Every packable solution project is discovered without a manual CI list.
- Old package identities have documented successors and an explicit unlisting path.
- Tale uses one package, namespace, and public type vocabulary.

## Delivered steps

| Step | Outcome |
|---|---|
| Publish gate and pack discovery | Release-triggered publishing, structural project packing, and manual unlisting workflow. |
| NuGet metadata | Shared license, icon, readme, repository, and SourceLink settings. |
| Logging rename | Logging extensions and dependent API/HTTP callers moved to `Sol` names. |
| Data and transport rename | CQRS, HTTP, MessageBus, Cache, SQL, and Hangfire public registration names updated. |
| API and Authentication rename | Versioning and authentication installers normalized. |
| Tale rebrand | New package, namespace collapse, public type rename, route update, tests, and DreamTravel migration. |
| Authentication cleanup | Internal service-provider construction removed and host tests added. |
| Deprecated projects | Scheduler and Guards marked obsolete and non-packable. |
| Version flip | Shared 1.0 package line activated with the Logging exception. |
| Package discovery docs | README successors and package parity updated. |
| Release operations | Release runbook, CI/CD docs, ghost-package inventory, and unlisting instructions. |
| Documentation sweep | Tale docs, coding instructions, skill names, diagrams, and deprecated landing pages updated. |

## Expected consequences

### Positive

- Consumers receive one predictable registration vocabulary.
- New modules cannot be silently omitted from structural packing.
- Package versions and metadata are centrally governed.
- HTTP and Tale have explicit predecessor/successor stories.
- The orchestration package presents one Tale vocabulary.

### Negative

- Every supported package consumer takes compile-time breaks when upgrading.
- The Tale migration changes package ID, namespace, and public types without forwarders.
- Ghost package identities require continued retirement stewardship.
- The repository can contain renamed source before a deliberate release publishes it.

## Completion summary

All planned waves shipped after the premortem mitigations. The package versions, Tale rebrand, API
renames, dynamic packing, gated publishing, centralized metadata, deprecation changes, release
runbook, public package parity, and manual unlisting workflow were delivered. Builds and affected
test suites remained green through the rename waves.

Current policy lives in
[`../architecture/package-release.md`](../architecture/package-release.md),
[`../architecture/naming-and-public-api.md`](../architecture/naming-and-public-api.md), and
[`../architecture/tale-framework.md`](../architecture/tale-framework.md).

## Deviations

- The historical record remained marked proposed after implementation.
- The documentation sweep was incomplete; several user docs retained removed API names.
- Current packable project count and unlisting inventory differ from planning snapshots.
- Tale manager/controller operation names retained their historical `Story` suffix, and the Tale ID
	prefix remained `STR`; public types and package naming still use Tale.
- Deprecated docs remained migration landing pages instead of being deleted.
- The migration guide, changelog, and per-project readmes were cut from pre-1.0 scope.
- NuGet server-side deprecation was replaced by automated unlisting because the assumed CLI command
	did not exist.
- Historical decision records retained their original Story terminology instead of being
	mechanically rewritten.

## Follow-ups

- Keep public docs aligned with source extension methods in every API change.
- Derive release package inventory from the solution rather than copying package counts into docs.
- Treat any future public rename as a major-version decision.
